# ケアシフト V3 | 重度訪問介護事業向け業務効率化 SaaS

![ケアシフト](https://gyazo.com/e2bc82e23c954f42a37a31e661fb9d25.jpg)

**「ケアシフト」** は、重度訪問介護事業向けの業務効率化 SaaS です。  
前職の訪問介護事業所で管理業務を担当していた際、既存 SaaS の操作が複雑で現場に定着しづらいという課題を感じたことをきっかけに開発しました。

訪問介護ではシフト管理・勤怠管理・連絡共有など多くの業務が発生しますが、これらが複数のツールに分散しているケースも多く、管理負担の増加やミスの原因となっていました。
---

# なぜこのツールを開発するのか
## 前職の管理業務で感じた課題

## 煩雑化するシフト管理

現状、訪問介護のシフト管理は表計算ソフトが主流です。  
前職でも従業員1人につき複数行のレコードで管理していたため、人数増加に比例してデータ量が増大し、可読性が低下していました。

その結果

- 重複登録  
- 夜勤明けに日勤を入れてしまう誤配置  
- シフトの入れ忘れ  

といったミスが増え、毎月のシフト作成に多くの時間がかかっていました。

GAS による機能追加や有料の管理ソフト導入なども試しましたが、  
スプレッドシートでは細かな権限管理が難しく、既存 SaaS も操作が複雑で現場に定着せず、  
最終的には表計算ソフトによる管理に戻ってしまう状況でした。

---

## 直行直帰の事業形態のため、出勤状況を把握しづらい

訪問介護では直行直帰の働き方が一般的であるため、管理者がリアルタイムで出勤状況を把握することが難しいという課題がありました。

前職では勤怠管理を LINE で行っていましたが、人数が増えるにつれて

- 誰が出勤済みなのか  
- 誰が遅れているのか  

といった状況把握が難しくなりました。

その結果、支援開始時間を過ぎてから利用者様から連絡が入るケースも発生していました。

---

## 訪問先が毎日変わることで発生するケアレスミス

訪問介護では日々の訪問先が変わるため、確認不足によるケアレスミスが発生しやすい環境でした。

具体的には

- 訪問先を取り違えて別の場所へ向かってしまう  
- 誤った住所を共有してしまう  

といったミスがあり、支援開始の遅れや関係者への連絡負担につながっていました。

---

## 特定事業所加算に必要なチャット機能

訪問介護では、一定条件を満たした事業所に対して介護報酬の加算制度があります。

その条件の一つに

> 「事業所の指揮命令系統に基づき、組織的かつ計画的にサービス提供が行われている体制」

という要件があり、これを満たすためには私的な連絡手段ではない業務用のコミュニケーションツールが必要になります。

前職では、この目的のためにチャット機能を持つ有料アプリケーションを導入していました。

# 解決策

ケアシフトは上記の課題に対し、以下の解決策を提供します。

## 1. 最大限シンプルにしたシフト作成・共有機能

訪問介護の管理職は PC 操作が得意な方ばかりではありません。  
そのため選択肢を必要最小限にし、直感的に操作できる UI を目指しました。

少ないクリック数でシフト作成ができ、管理者は入力作業ではなく人員配置の判断に集中できます。

---

## 2. 一目で分かる勤怠管理機能

1画面で利用者ごとの勤怠状況を確認できます。

出勤 / 未出勤を色分けして表示することで、状況把握にかかる時間を削減します。

---

## 3. ワンタップ経路表示

利用者の住所を事前登録しておくことで、当日はマップボタンを押すだけで現在地から訪問先までナビゲーションできます。

住所伝達ミスや訪問先の取り違えといったケアレスミスを防ぎます。

---

## 4. アプリ内リアルタイムチャット

私用の連絡手段とは切り分けたアプリ内チャットでリアルタイムに連絡できます。

---

これらの機能により、日々の煩雑な管理業務を効率化し  
**月間100時間の業務削減**を目指しています。

---

# 新規ユーザー獲得方法

新規事業では顧客接点を増やすことが重要だと考え、対面営業を行いました。

営業用のパンフレットを作成し、実際に企業へ訪問してヒアリングを実施しました。

その結果
- 重度訪問介護だけでなく訪問介護にも対応してほしい  

といったフィードバックをいただきました。

現在はこれらの要望を反映した改善を実装しており、対応完了後に再度営業を行う予定です。

# 技術構成

## バックエンド
**Ruby / Ruby on Rails**

Rails の「設定より規約」の思想に基づき、規約に沿った実装を行うことで少ないコード量で保守性の高いサーバーを構築できる点に魅力を感じ、Rails を採用しました。

主な実装内容

- Devise / OmniAuth を利用した認証機能
- 二段階認証（2FA）
- Pundit による権限管理
- 各責務や権限ごとにディレクトリを分離し、可読性の高い設計としています。

テストには **RSpec / FactoryBot / Capybara** を利用し、  
**SimpleCov によるコードカバレッジ 80％以上** を維持しています。

---

## フロントエンド
**React / TypeScript / Vite**

将来的にモバイルアプリ展開を視野に入れ、React Native への展開可能性を考慮して React を採用しました。

v2 では Rails のサーバーサイドレンダリング中心の構成でしたが、  
v3 では **フロントエンド分離構成（SPA）** に移行しました。

これにより

- 仮想 DOM による高速な UI 更新
- CDN キャッシュの活用
- UX の向上

を実現しています。

---

## データベース
**MySQL / Amazon RDS**

全テーブルに **Office 外部キー** を持たせた **マルチテナント構造** を採用しています。

将来的に大規模顧客の利用が発生した場合でも、テナント単位でのデータ分離やスケール対応が行いやすい設計としています。

---

## インフラ
**AWS（EC2 / ALB / S3 / CloudFront / DynamoDB / SSM）**

インフラは AWS 上に構築しています。

主な構成

- EC2 : アプリケーションサーバー
- ALB : 負荷分散
- CloudFront : CDN
- S3 : 静的アセット / ActiveStorage
- DynamoDB : Terraform State Lock
- SSM : シークレット管理

CI およびローカル環境からは **AssumeRole** を利用し、  
認証情報をコードや環境変数に持たない **シークレットレス運用** を意識しています。

---

## プロビジョニングツール
**Terraform**

将来的なチーム開発を見据え、v3 から **IaC（Infrastructure as Code）** を導入しました。

Terraform State は

- **S3 に保存**
- **DynamoDB によるロック管理**

を行い、同時実行による衝突を防ぐ構成としています。

---

## CI / CD
**GitHub Actions**

CI では

- テスト実行
- 静的解析
- セキュリティチェック

を自動化しています。

Terraform は

- **Pull Request 作成時 : plan**
- **main マージ時 : apply**

が実行される構成としています。
